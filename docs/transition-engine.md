# Smart Transition Engine (TEA)

Архитектура модуля выбора и исполнения переходов между треками в BPMind Auto Mixer.

## Контекст

В проекте уже есть:

- сканирование музыкальной библиотеки;
- анализ BPM, громкости, energy map, точек перехода, groove;
- построение последовательности треков (**Mix Builder**).

Следующий этап — отдельный модуль, который решает **как** стыковать соседние треки, а не только **в каком порядке** их играть.

## Главная идея

Это **не** набор случайных эффектов для красоты.

Цель — имитировать решение опытного DJ перед каждой стыковкой:

> «Какой переход здесь прозвучит лучше всего?»

Для каждой пары **Track A → Track B** движок определяет:

- какой тип перехода (или цепочку) использовать;
- стоит ли вообще применять эффект;
- насколько плавным должен быть переход;
- какой сценарий сведения выбрать.

**Базовая формула продукта:**

```text
transition = decide(context)   # умный режим (по умолчанию)
```

Не:

```text
transition = random()          # только в режиме отладки
```

---

## Пайплайн

```text
Library Scan
    ↓
Track Analysis
    ↓
Mix Builder          ← только порядок треков, play_from / play_until
    ↓
Transition Engine    ← выбор и план переходов (TEA)
    ↓
Audio Engine         ← DSP: blend, filter, echo, …
    ↓
Playback / Export
```

**Mix Builder** и **Transition Engine** — два независимых этапа.  
Mix Builder не должен зашивать тип перехода навсегда; TEA получает готовую сессию и возвращает план стыковок.

---

## Режимы выбора переходов

Один и тот же набор **Transition Profiles** (профилей эффектов), но **три стратегии** того, как их назначать на стыковки.

### 1. Auto (Smart) — режим по умолчанию

| | |
|---|---|
| **Кто видит** | Обычный пользователь |
| **Где в UI** | Нигде явно не выбирается; работает «из коробки» |
| **Поведение** | TEA анализирует контекст каждой пары A→B и вызывает `decide(context)` |
| **Философия** | Пользователь **не** крутит эффекты вручную |

Это основной продуктовый режим BPMind.

### 2. Fixed — один профиль на весь сет

| | |
|---|---|
| **Кто видит** | Разработка, отладка, продвинутый CLI |
| **Где** | CLI (`--transition smooth_blend`), опционально «Дополнительно» / dev-флаги |
| **Поведение** | Все стыковки в сете получают **один** выбранный тип (например, только Filter Sweep) |
| **Зачем** | Сравнить звучание профиля, тесты, демо одного эффекта |

Не предназначен для рядового сценария «собрать сет на вечеринку».

### 3. Random — случайный профиль на каждую стыковку

| | |
|---|---|
| **Кто видит** | Только отладка / CLI |
| **Где** | CLI (`--transition-mode random`), не на главном экране GUI |
| **Поведение** | На каждой стыковке `random.choice(profiles)` без анализа контекста |
| **Зачем** | Быстро проверить, что все DSP-профили рендерятся и не ломают плеер |

**Важно:** random в продукте **не** заменяет TEA. Это тупой режим для разработки, не «умный random по вайбу».

### Сводка

| Режим | Назначение | Для пользователя |
|-------|------------|------------------|
| **auto** (TEA) | Музыкально осмысленный выбор | Да, по умолчанию |
| **fixed** | Один эффект на весь сет | Нет (debug / CLI) |
| **random** | Случайный эффект на каждую стыковку | Нет (debug / CLI) |

Выбранный режим и результат планирования **сохраняются в рецепте сета** (`PlannedTransition`), чтобы воспроизведение и экспорт совпадали.

## UX vs отладка

### Что видит обычный пользователь (UX)

- Скан → построить микс → play / сохранить сет / экспорт.
- Переходы подбираются **автоматически** (режим **auto**).
- В интерфейсе **нет** списка эффектов, random-переходов и ручного выбора «echo на трек 3».
- Опционально: ненавязчивая подсказка вроде «Переходы: автоматически» (без деталей профиля).

Визуализация типа перехода (иконка, название на таймлайне) — кандидат на **v2.0 UI**, не обязательна для MVP TEA.

### Что остаётся для отладки

- CLI-флаги: `--transition-mode auto|fixed|random`, `--transition <profile_id>`.
- Фиксированный профиль на весь сет (**fixed**).
- Случайные профили по стыковкам (**random**).
- Логирование решений TEA: context → выбранный профиль → score / причина.
- Возможность отключить отдельные профили или cooldown в тестах.

GUI: только за флагом «Дополнительно» или отдельной dev-сборки — **не** на главном потоке v1.x.

## Transition Context

Перед выбором эффекта TEA формирует объект контекста для пары **A → B**.

Минимальный набор (MVP):

| Поле | Источник в BPMind |
|------|-------------------|
| BPM A / B, Δ BPM | quick / deep analysis |
| Громкость A / B | analysis |
| Энергия хвоста A / головы B | `energy_map` |
| Groove-совместимость пары | `groove_similarity` |
| Тип окончания A (quiet, drop, …) | `transition_candidates` |
| Длина outro / характер intro | эвристики по energy + bounds |
| История последних профилей | cooldown tracker в рамках сета |

Расширения (не MVP): тональность, дроп, брейкдаун, спектр, вокал, бас, оценки пользователя.

## Transition Profiles

Каждый переход — отдельный **профиль** с метаданными:

- название и id;
- описание;
- когда применять / когда **не** применять;
- насколько заметен эффект;
- приоритет при scoring;
- ограничения (мин/макс длительность, только при Δ BPM < N);
- **cooldown** (не чаще N раз за сет или M стыковок подряд).

Профиль сам «знает», для каких ситуаций он подходит; TEA собирает score по всем подходящим и выбирает лучший с учётом cooldown.

### Профили первой версии (целевой набор)

| ID | Название | Назначение (черновик) |
|----|----------|------------------------|
| `smooth_blend` | Smooth Blend | Стандартное плавное сведение; чаще всего |
| `echo_out` | Echo Out | Растворение хвоста; сложные смены |
| `filter_sweep` | Filter Sweep | LP/HP перед вводом следующего трека |
| `bass_swap` | Bass Swap | При близком BPM, смена басовой линии |
| `reverse_reverb` | Reverse Reverb | Перед новым вступлением |
| `vinyl_brake` | Vinyl Brake | Редкий художественный эффект |
| `tape_stop` | Tape Stop | Резкая остановка предыдущего |
| `impact` | Impact Transition | Перед энергичным вступлением |

MVP реализации: начать с **2–3** профилей (smooth_blend + filter_sweep + echo_out), остальные — итеративно.

Правила «когда применять» — живая таблица; стартовые эвристики по Δ BPM, energy, groove (см. обсуждение v1.3).

### Cooldown

Чтобы не было шаблона «каждый трек — tape stop»:

- `echo_out`, `tape_stop`, `vinyl_brake`, `impact` — повышенный cooldown;
- `smooth_blend` — низкий cooldown, высокий приоритет по умолчанию;
- в контекст передаётся `recent_profiles[]` за последние K стыковок.

### Составные переходы (post-MVP)

Переход может быть **цепочкой**, например:

```text
echo_out → filter_sweep → smooth_blend
```

Компоновщик (`composer`) строит сценарий; в MVP достаточно **одного** профиля на стыковку.

## Архитектура кода (целевая)

```text
engine/transitions/
  context.py           # TransitionContext, сбор из Track A/B + история
  cooldown.py          # учёт недавних профилей
  profiles/            # SmoothBlend, EchoOut, …
  registry.py          # реестр профилей, scoring
  planner.py           # SmartTransitionEngine.plan(session, tracks, mode)
  modes.py             # auto | fixed | random
  audio/               # DSP-исполнение (blend, filter, echo, …)
  crossfade.py         # (существующий smooth blend)
```

**Поток:**

```python
session = mix_builder.generate(...)  # без типов переходов или с placeholder
plan = transition_engine.plan(
    session,
    tracks_by_id,
    mode=TransitionMode.AUTO,       # default
    fixed_profile=None,             # только для mode=FIXED
)
session = session.with_transitions(plan)
```

**Модель данных:** расширить `PlannedTransition`:

- `profile_id: str` (или `chain: list[str]` позже);
- `crossfade_duration_sec` / `duration_sec`;
- опционально `metadata` для отладки (score, причина выбора).

Плеер и экспорт вызывают **единый** `apply_transition(profile_id, outgoing, incoming, params)`.

## Философия

BPMind не должен выглядеть как программа, которая **случайно** вставляет красивые эффекты.

В продукте пользователь доверяет **auto (TEA)**.  
Режимы **fixed** и **random** — инструменты разработчика, чтобы проверить, что все профили работают, а не альтернатива «умному» режиму для конечного пользователя.
